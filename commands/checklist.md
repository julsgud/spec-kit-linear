# Checklist

You are generating a requirements quality checklist for a Linear issue's specification. This checklist validates whether the specification is complete, clear, consistent, and measurable. It asks "Is X specified?" — never "Does X work?"

## Input

The user provides a Linear issue URL or identifier as `{{ args }}`. If no argument is provided, the issue is inferred from the current git branch name.

## Step 1: Parse the issue identifier

**If `{{ args }}` is provided**, extract the issue identifier from the input:
- `https://linear.app/{workspace}/issue/{ID}/{slug}` → use `{ID}`
- `https://linear.app/{workspace}/issue/{ID}` → use `{ID}`
- `BOT-140` (bare identifier) → use directly

If the input does not match any of these formats, stop with:
```
Error: Could not parse Linear issue from "{{ args }}".
Expected a Linear URL (https://linear.app/.../issue/BOT-140/...) or identifier (BOT-140).
```

**If `{{ args }}` is empty**, infer the issue from the current git branch:
1. Run `git branch --show-current` to get the branch name.
2. If the result is empty (detached HEAD), stop with:
   ```
   Error: Cannot infer issue — HEAD is detached. Provide an issue identifier explicitly.
   ```
3. If the branch is `main`, `master`, or `develop`, stop with:
   ```
   Error: Cannot infer issue from default branch "{branch}". Provide an issue identifier explicitly.
   ```
4. Extract the first match of the pattern `[A-Za-z]+-[0-9]+` from the branch name and uppercase it to get the issue identifier.
5. If no match is found, stop with:
   ```
   Error: Cannot infer issue from branch "{branch}" — no identifier found. Provide an issue identifier explicitly.
   ```
6. Print: `Inferred issue {ID} from branch "{branch}".`

## Step 2: Read the issue and all artifact comments

Call `get_issue` with the extracted identifier.

Call `list_comments` and collect all spec-kit artifact comments by searching for sentinels:
- `<!-- speckit:specification:v* -->` — Specification (**required**)
- `<!-- speckit:clarifications:v* -->` — Clarifications (optional)
- `<!-- speckit:analysis:v* -->` — Analysis (optional)
- `<!-- speckit:plan:v* -->` — Plan (optional)

For each artifact type, use the comment with the highest version number.

If no specification comment is found, stop with:
```
Error: No Specification found on {ID}.
Run /speckit.linear.specify {ID} first.
```

## Step 3: Check for existing checklists

Search the comments for existing checklist sentinels (`<!-- speckit:checklist:v* -->`). If previous versions exist, note the highest version number.

## Step 4: Generate checklist items

Using the specification (and clarifications/analysis/plan if present), generate checklist items across five validation dimensions. Target 15–30 items total (fewer for tight specs, more for loose ones).

Every item must be specific to the actual specification content — no generic boilerplate. Each item asks whether something is specified, defined, or documented.

**Item format:** `- [ ] CHK### - Question about requirement quality? [Dimension, Ref]`

Traceability refs point to spec sections (`Goals-1`, `AC-3`, `US-2`, `Scope`, `Tech`) or plan tasks (`T001`).

### Completeness
Are all necessary requirements present? Check for missing goals, user stories, acceptance criteria, scope boundaries, and technical details.

### Clarity
Are requirements unambiguous and specific? Flag vague language ("fast", "easy", "intuitive"), undefined terms, and requirements that could be interpreted multiple ways.

### Consistency
Do requirements align without contradictions? Check that goals match acceptance criteria, scope doesn't conflict with user stories, and clarification findings are resolved.

### Measurability
Can requirements be objectively verified? Check that acceptance criteria have concrete pass/fail conditions, performance targets have numbers, and "done" is well-defined.

### Coverage
Are all scenarios, edge cases, and roles addressed? Check error handling, boundary conditions, permissions, and whether every plan task traces back to a requirement.

**Additional rules:**
- If clarifications or analysis exist, check whether their findings are resolved in the latest specification.
- If a plan exists, verify each task traces to a requirement.

## Step 5: Post the checklist comment

Determine the version number:
- If no previous checklist comment exists → `v1`
- If previous versions exist → increment

Format the comment body as:

```markdown
<!-- speckit:checklist:v{N} -->
## Checklist

> Generated by spec-kit-linear | v{N} | {today's date}

### Completeness
- [ ] CHK001 - ...? [Completeness, Goals-1]
- [ ] CHK002 - ...? [Completeness, AC-3]

### Clarity
- [ ] CHK003 - ...? [Clarity, US-2]

### Consistency
- [ ] CHK004 - ...? [Consistency, Scope]

### Measurability
- [ ] CHK005 - ...? [Measurability, AC-1]

### Coverage
- [ ] CHK006 - ...? [Coverage, T001]
```

Call `save_comment` to post this as a new comment on the issue.

If a previous version exists, edit the previous version's comment to prepend `**(superseded by v{N})**` on the line immediately after the `## Checklist` header.

## Output

Summarize what was done:
- Issue: `{ID} — {title}`
- Artifacts used: list which artifacts informed the checklist
- Checklist version: `v{N}`
- Item counts: {count} per dimension, {total} total
